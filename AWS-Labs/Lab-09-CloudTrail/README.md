# Lab 09: CloudTrail Auditing

## Objective

Record AWS API activity with a multi-Region CloudTrail and deliver audit logs to S3.

## AWS Services Used

AWS CloudTrail, Amazon S3, and AWS IAM.

## Steps Performed

1. Created the `devsecops-trail` multi-Region trail.
2. Enabled global service events and active logging.
3. Applied an S3 bucket policy for CloudTrail delivery.
4. Performed controlled API activity.
5. Verified event history and S3 log delivery.

```bash
aws cloudtrail get-trail-status --name devsecops-trail
aws cloudtrail lookup-events --max-results 10
aws s3 ls s3://<trail-bucket>/AWSLogs/ --recursive
```

## Architecture Explanation

CloudTrail records management events generated by console, CLI, SDK, and service activity. Logs are delivered to S3 for durable review and investigation.

## Security Considerations

- Enable encryption, log-file validation, retention, and restricted bucket access.
- Protect trails from disabling or deletion.
- Alert on sensitive events such as policy changes, root usage, and logging changes.
- Centralize audit logs in a separate security account for production.

## Validation Steps

Confirm logging is active, the trail is multi-Region, recent events appear in Event History, and compressed log files reach the correct S3 prefix.

## Learning Outcomes

- Used CloudTrail to answer who, what, when, and where for API activity.
- Connected audit configuration with durable S3 storage.
- Identified controls needed to protect audit evidence.
